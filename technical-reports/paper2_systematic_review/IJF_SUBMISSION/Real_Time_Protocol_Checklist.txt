═══════════════════════════════════════════════════════════════════
                   REAL-TIME PROTOCOL AUDIT CHECKLIST
                For Sentiment-Based Forecasting Systems
═══════════════════════════════════════════════════════════════════

PROBLEM: ~35% of sentiment forecasting papers violate real-time 
data discipline, using information not available at forecast time. 
This checklist helps practitioners and auditors verify compliance.

═══════════════════════════════════════════════════════════════════

SECTION 1: DATA SOURCES - Is sentiment constructed only from 
data available at forecast time?

□ News sentiment: Published before forecast date?
   Example: For GDP forecast on June 1, use news published by May 31
   ✓ OK: Wall Street Journal articles published May 31
   ✗ NOT OK: CNN breaking news published June 1 at 2pm

□ Social media sentiment: Data collected before forecast?
   ✓ OK: Twitter/Reddit data collected by 11:59pm previous day
   ✗ NOT OK: Real-time tweets from forecast day included

□ Survey sentiment: Release date vs. survey date clear?
   ✓ OK: Consumer Confidence Index from Conference Board
       released on ~last Tuesday, covers month before
   ✗ NOT OK: Using survey respondent data from forecast month

□ Central bank communication: Timing explicit?
   ✓ OK: FOMC statements analyzed; release time documented
   ✗ NOT OK: Fed officials' remarks later contradicted by official statement

□ Proprietary data: Collection methodology transparent?
   ✓ OK: "We scraped X news sources daily at 6am UTC"
   ✗ NOT OK: "We use proprietary sentiment scores" (can't verify timing)

═══════════════════════════════════════════════════════════════════

SECTION 2: MODEL SPECIFICATION - Are hyperparameters chosen 
without access to test data?

□ Hyperparameter tuning: Cross-validation on training data only?
   ✓ OK: "We used 5-fold CV on 2007-2015 data"
   ✗ NOT OK: "We selected parameters on full 2007-2019 dataset"

□ Threshold selection: Optimized on training data?
   ✓ OK: "Dictionary threshold for positive/negative set on 2007-2015"
   ✗ NOT OK: "Threshold optimized on test data to maximize accuracy"

□ Feature engineering: No information leakage?
   ✓ OK: "Dictionary terms pre-defined; not changed after analysis"
   ✗ NOT OK: "Added negative words to dictionary after seeing test results"

□ Model selection: Chosen before test evaluation?
   ✓ OK: "Selected Random Forest; evaluated on hold-out 2019 data"
   ✗ NOT OK: "Tried 10 models; reported best performer"

═══════════════════════════════════════════════════════════════════

SECTION 3: EVALUATION PROCEDURE - Is test set truly unseen?

□ Train/test split: Temporally ordered?
   ✓ OK: Train 2007-2015, Test 2016-2019 (no look-ahead)
   ✗ NOT OK: Random 80/20 split (data leakage across time)

□ Rolling window: Expanding or fixed window used?
   ✓ OK: Train on 2007-2015, test 2016; then train 2007-2016, test 2017
   ✗ NOT OK: Train on entire dataset, slide window for testing

□ Multi-step forecasting: Future values not used?
   ✓ OK: One-step-ahead or explicit multi-step (use only past data)
   ✗ NOT OK: Using target variable from forecast month in sentiment

□ In-sample fit: Not reported as out-of-sample?
   ✓ OK: "OOS R² = 0.12; in-sample R² = 0.45"
   ✗ NOT OK: Reports only in-sample performance as "validation"

═══════════════════════════════════════════════════════════════════

SECTION 4: DATA PUBLICATION - Can results be reproduced?

□ Sentiment data: Made available or reproducibly constructed?
   ✓ OK: "Sentiment scores available at https://github.com/..."
   ✗ NOT OK: "Proprietary; available upon request"

□ Sentiment construction: Methods documented?
   ✓ OK: "Dictionary from Loughran-McDonald (2011); available online"
   ✗ NOT OK: "Custom dictionary; details available upon request"

□ Code: Repository with timestamps?
   ✓ OK: "Code at https://github.com/repo; data from 2019-06-01"
   ✗ NOT OK: No code provided

□ Baseline models: Clearly specified?
   ✓ OK: "Baseline: ARIMA(1,1,1) estimated on training data only"
   ✗ NOT OK: "Compared to baseline" (specification unclear)

═══════════════════════════════════════════════════════════════════

SECTION 5: DOCUMENTATION - Is transparency sufficient?

□ Explicit real-time statement: Paper states RT protocol?
   ✓ OK: "We follow real-time data protocol: only information 
          available at forecast date enters models"
   ✗ NOT OK: No mention of real-time discipline

□ Sentiment timing: Data sources and release dates clear?
   ✓ OK: "WSJ articles published on forecast date - 1; we download 
          daily at 6am UTC"
   ✗ NOT OK: "We use news sentiment" (when collected? unclear)

□ Model update frequency: Retraining schedule specified?
   ✓ OK: "Model estimated on rolling 5-year window; retrained monthly"
   ✗ NOT OK: "Model updated dynamically" (no schedule)

□ Limitations acknowledged: Papers mention potential RT violations?
   ✓ OK: "Limitation: Our analysis assumes real-time data; 
          actual deployment requires careful engineering"
   ✗ NOT OK: No discussion of real-time challenges

═══════════════════════════════════════════════════════════════════

SCORING YOUR SYSTEM

Count YES answers:

16-20: EXCELLENT - Rigorous real-time protocol
12-15: GOOD - Minor gaps; acceptable for deployment
8-11:  FAIR - Significant gaps; needs improvement before production
5-7:   POOR - Major violations; not ready for real-time deployment
<5:    CRITICAL - Likely look-ahead bias; do not deploy

═══════════════════════════════════════════════════════════════════

REMEDIATION GUIDE

If you score <12, fix these in priority order:

1. CRITICAL (do first):
   • Implement rolling window validation (removes most bias)
   • Document exact data timing (when do you observe sentiment?)
   • Separate hyperparameter tuning to training data only

2. IMPORTANT (do next):
   • Publish sentiment data or code for reproducibility
   • Add explicit real-time statement to paper/documentation
   • Create data availability statement

3. NICE-TO-HAVE (polish):
   • Provide code repository with full reproducibility
   • Include sensitivity analysis on data timing assumptions
   • Compare to alternative sentiment measures

EXAMPLE: APPLYING CHECKLIST

Paper: "Forecasting GDP with News Sentiment (2007-2019)"

Verdict: FAILS real-time protocol
Reasons:
  ✗ "We use news published on forecast month" (future data!)
  ✗ "Hyperparameters optimized on full 2007-2019 dataset"
  ✗ No rolling window; single 80/20 train/test split
  ✗ In-sample R² of 0.35 reported; OOS R² not mentioned

Recommendation: Do NOT use for real-time forecasting without 
fixing: (1) exclude forecast-month news, (2) CV on training 
data only, (3) rolling window evaluation, (4) report OOS only
